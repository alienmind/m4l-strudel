# The Sample Browser (`ableton-sample-browser`)

## Overview
The **Sample Browser** is a generic browser for open-source samples available in the market. It allows users to:
- Navigate and search for samples across different open-source projects.
- Pre-listen to samples perfectly synced to the Ableton Live transport (BPM and beat phase).
- Flag/favorite samples.
- Download them locally to use within a track.

Instead of trying to reinvent the capabilities of Ableton's own extremely powerful samplers (like Sampler or Simpler), this device focuses entirely on **discovery and acquisition**. It acts similarly to the Splice integration, but exists natively within the Max for Live ecosystem.

## Device Architecture: The "Floating Utility"

Because the core feature is browsing and downloading, the device does not behave like a standard instrument or effect. 

The `ableton-sample-browser` is designed as an **Audio Effect**. However, it is fundamentally a "Utility" device.
- **Placement**: Users are instructed to place it on their Master track, or on a dedicated "Preview" audio track.
- **DSP Footprint**: It does not process audio. It passes the audio signal through 100% unchanged (dry).
- **The UI**: The device itself has a minimal UI in the device view, primarily housing a button that spawns a large, **floating `[jweb]` window**. This floating window contains the actual web-based sample browser UI.

## Why this approach?

### 1. Transport Sync (LOM Access)
By existing as a Max for Live device inside the Live session, the device has native, instant access to the Live Object Model (LOM). It constantly knows the current BPM, the transport state (playing/stopped), and the exact beat position. 
When a user clicks "preview" on a loop in the browser, the web UI can play the audio perfectly quantized to the track's downbeat.

### 2. Audio Routing

> [!WARNING]
> **This section used to claim that audio generated by the floating `[jweb]` window comes
> out of the track. IT DOES NOT, and the whole preview feature was designed on top of
> that mistake.**
>
> **`[jweb]` has no `~` outlets.** Its Chromium process sends audio straight to the
> operating system's default output device - bypassing the track, the fader, the
> monitor cue and every effect after it. That is not a detail; it is the entire reason
> `m4l-jweb` carries a native-audio-bridge item on its roadmap (`FEAT-STRUDEL-002`).
> A WebAudio preview would play at full volume out of the user's default device while
> Live's meters sat silent.

So a preview that honours the user's monitoring **cannot be WebAudio in the browser
window**. It has to be a real signal in the Max patcher:

- the sample is **downloaded first** (`fetchToFile()` - shipped), then
- loaded into a `[buffer~]` and played by `[play~]`/`[groove~]` - the **`samples` chain**,
  which is the one dependency below that has not shipped yet.

Downloading before previewing is therefore not a detour to be optimised away. **It is
the only path to audio that Live can hear**, and it is the same rule the rest of this
architecture follows: bulk audio travels on disk, and Max plays it.

(The upside: a `buffer~` played from `[phasor~]` locked to `current_song_time` is
*sample-accurately* synced to the transport, which a WebAudio preview could never be.)

### 3. Drag-and-Drop Workflow
Once a user decides to download a sample, it is saved to a known directory on their local file system. The user can then navigate to that folder in Live's native "Places" browser and drag the WAV file directly into a Drum Rack, Simpler, or an Arrangement track.

## The Node.js Problem (And the `[maxurl]` Solution)

Historically, making HTTP requests or downloading files within Max required using `[node.script]`. This is considered a poor implementation because:
- It introduces a heavy dependency (Node.js).
- It is notoriously unstable and prone to crashing Ableton Live when fetching large files or making rapid requests.

**The Solution:**
This device will completely bypass Node.js by utilizing Max's native **`[maxurl]`** object. `[maxurl]` is a native C implementation that can handle HTTP GET/POST requests and features a "download to disk" capability.

## Dependencies on `m4l-jweb`

| Capability | State |
|---|---|
| **Floating window** (`FEAT-STRUDEL-001`) - the device view is a fixed ~169 px and does not scroll, so the browser lives in a window of its own | **SHIPPED** in 0.6.0: `windows: { browser: window({...}) }` in `surface.ts` + `useWindow()` |
| **Fetch-to-disk** - `[maxurl]`, no `[node.script]` | **SHIPPED** in 0.6.0: `fetchToFile(url, path, onProgress)`. It downloads to a temp path and only places the file once the response is validated, so a 404 cannot overwrite a sample you already had |
| **Transport-synced preview** - `[buffer~]` + `[play~]` in the patcher | **OPEN**: the `samples` chain, `m4l-jweb` Priority 1 item 2. This is now the only thing blocking the device |

Two of the three shipped in `m4l-jweb` 0.6.0. **The preview is the remaining blocker** -
and note from §2 above that it is a genuine one, not a nicety: without the `samples`
chain there is no way to make a sound the user's track can hear.
