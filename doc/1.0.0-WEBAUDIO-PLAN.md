# 1.0.0 WEBAUDIO REWRITE PLAN

## Executive Summary
The discovery of Max 9's `jweb~` object invalidates the fundamental technical limitation that shaped `m4l-jweb` and `m4l-strudel`. Previously, the embedded Chromium browser (`jweb`) lacked audio outlets, forcing a strict separation between the "Control Plane" (React/JS) and the "Audio Plane" (Max native DSP).

With `jweb~`, the browser's Web Audio API output can be routed **directly into Max's audio graph** and out to Ableton Live's track output. 

We can now drastically simplify both codebases, dropping complex workarounds (offline rendering loops, Max-side polyphony orchestration) in favor of native JavaScript audio generation.

---

## Part 1: `m4l-jweb` Framework Rewrite

### 1. Replace `[jweb]` with `[jweb~]`
The core patcher generation must be updated to use the new audio-capable browser object.
- **Device Type Branching:** For `instrument` (and potentially `audio`) devices, the build step will swap `maxclass: "jweb"` for `maxclass: "jweb~"`. 
- **Outlet Shifting:** The standard `jweb` has 1 outlet (messages). `jweb~` has 3 outlets: `[signal L, signal R, message]`.
- **Wiring Updates:** The bridge logic (e.g., `claimAppMessages` in `chains.mjs`) must be updated to read the message stream from `[jweb~, 2]` instead of `[jweb, 0]`.

### 2. The `webaudio` Chain
We will introduce a new base chain: `webaudio`.
- This chain claims the first stage of the signal path.
- It wires `jweb~` signal outlet 0 (Left) and 1 (Right) into the active audio graph (`ctx.setAudioOut`).
- This allows developers to just output audio in the browser, and the framework automatically pipes it to `plugout~`.

### 3. Modules to Drop (The Great Deletion)
The framework currently maintains thousands of lines of code to orchestrate audio entirely on the Max side. These are now obsolete:
- **Drop `renderplay` chain:** The "superdough" double-buffered WAV rendering loop is completely unnecessary. Native Web Audio has lower latency and requires zero disk I/O.
- **Drop `samples` and `instrument` chains:** Managing Max `[buffer~]` loading and `[poly~]` voice allocation via messages is no longer needed. A developer can just use Tone.js, Strudel, or standard Web Audio `AudioBufferSourceNode` directly in React.
- **Bridge Protocol Cleanup:** Remove selectors like `render_load`, `render_arm`, `render_ready`, and the `fetchToFile` dependency for audio playback.

### 4. What we Keep
- **Control & Sync Bridge:** We still desperately need the `m4l-jweb` bridge for MIDI I/O (`notein`/`noteout`), Live parameter automation (`surface.ts`), Push integration, state saving (`useStateSync`), and transport polling (`tick`, `tempo`).
- **Post-Processing Chains:** Max-side DSP chains like `lowpass`, `drive`, and `crush` can be kept. They will simply process the signal *coming out* of `jweb~`, acting as zero-latency post-fx.

---

## Part 2: `m4l-strudel` Rewrite

### 1. Delete the Offline Render Loop
- **Drop `src/lib/render/offline.ts`:** Delete the entire system that records Strudel patterns into WAV files in the background.
- **Drop `saveToFile` integration:** We no longer need to write these WAVs to the Ableton project folder and send file paths to Max.
- **Simplify State:** Remove the double-buffering React state that tracks which Max `[groove~]` slot is currently playing and which is loading.

### 2. Enable Native Strudel Audio
- Boot Strudel using its standard Web Audio engine (e.g., `webaudio` output mode) directly in the React app.
- Audio generated by Strudel will natively stream out of `jweb~`.

### 3. Transport Synchronization
- Instead of relying on a Max `Task` to hard-sync a `[groove~]` loop boundary, we will feed the existing `tick` and `tempo` messages from the `m4l-jweb` bridge directly into Strudel's internal clock.
- When Ableton plays, Strudel plays. When Ableton's BPM changes, Strudel's Web Audio scheduler adjusts automatically.

### 4. UI/UX Improvements
- **Zero Latency Workflow:** Without the ~50ms+ latency of rendering a WAV to disk and swapping buffers, live coding will feel instantaneous. 
- **Remove Rendering Indicators:** Drop the visual progress bars and loading spinners associated with the `superdough` rendering pipeline.

---

## Implementation Sequence
1. **Update `m4l-jweb` Base:** Modify `packages/build/templates/base.json` and `chains.mjs` to emit `jweb~`. Fix the outlet indices (messages on outlet 2).
2. **Create `webaudio` Chain:** Wire the signal outlets to `plugout~`. Test with a simple HTML `<audio>` tag or `OscillatorNode`.
3. **Prune `m4l-jweb`:** Delete `renderplay` and `instrument` chains. Clean up `protocol.ts` templates.
4. **Strip `m4l-strudel`:** Rip out `offline.ts` and the double-buffering logic.
5. **Integrate Strudel Web Audio:** Wire Strudel's audio context to the browser output and hook up the Ableton transport bridge to its clock.
